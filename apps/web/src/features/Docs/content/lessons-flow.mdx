## Overview

The lesson flow is the core learning experience. A user enters a lesson, progresses through a sequence of exercises, submits answers, receives feedback, and upon completing all exercises, submits the full lesson for grading.

The lesson feature lives in `features/Lesson/` and includes the page, exercise templates, context, and several hooks that manage the state machine.

## Exercise Types

Every exercise has content **blocks** (educational material) and an optional **interaction** (the answerable element).

### Blocks

Blocks make up the instructional body of an exercise:

| Block Type  | Content                                              |
| ----------- | ---------------------------------------------------- |
| `header`    | Section heading text                                 |
| `paragraph` | Explanatory text                                     |
| `code`      | Syntax-highlighted code snippet (language + content) |
| `media`     | Image with alt text                                  |

The `BlockRenderer` component in the design system renders each block by type.

### Interaction Types

| Type       | Description                                                      |
| ---------- | ---------------------------------------------------------------- |
| **SELECT** | Multiple choice — pick the correct answer from a list of options |
| **CLOZE**  | Fill-in-the-gap — drag options into blanks within a code snippet |
| **INFO**   | No interaction — purely instructional (read and continue)        |

```ts
export type ExerciseInteraction =
  | { type: "SELECT"; items: string[]; correctValue: string }
  | {
      type: "CLOZE";
      file: InteractionFile;
      blanks: InteractionBlank[];
      options: string[];
    };
```

For `CLOZE` exercises, the `LudoCodePreview` widget splits the code content on `___` markers and renders each gap as an `OptionInputSlot`.

## The 3 Phases of an Exercise

Each exercise moves through a strict lifecycle:

### 1. Input Phase

The user interacts with the exercise by selecting options. Nothing is evaluated yet. Selections are stored in a local `useState` array of `AnswerToken`, allowing the user to revise, remove, or reorder choices freely.

The submit button stays disabled until the selection meets the requirements of a complete answer.

### 2. Staged Phase

The user clicks **Submit**. Input is now locked and the answer is evaluated.

The `useStagedAttempt` hook generates an `ExerciseAttempt` from the current selection:

```ts
export type ExerciseAttempt = {
  exerciseId: string;
  isCorrect: boolean;
  answer: AnswerToken[];
};
```

Visual feedback is shown — green for correct, red for incorrect.

### 3. Committed Phase

The user clicks **Continue** after seeing feedback:

- The staged attempt is merged into `committedExerciseSubmissions`
- If **incorrect**: the exercise resets for retry
- If **correct**: the next exercise unlocks
- When the **final exercise** is committed, all submissions are rolled up into a `LessonSubmissionRequest`

## Submission Structure

```ts
export type LessonSubmissionRequest = {
  submissionId: string; // Unique per attempt (prevents duplicates)
  lessonId: string;
  courseId: string;
  exercises: ExerciseSubmissionRequest[];
};

export type ExerciseSubmissionRequest = {
  exerciseId: string;
  version: number;
  attempts: ExerciseAnswer[];
};

export type ExerciseAnswer =
  | { type: "SELECT"; pickedValue: string }
  | { type: "CLOZE"; valuesByBlank: string[] };
```

Each exercise records all attempts (including wrong ones), allowing the backend to calculate accuracy.

## Module Navigation

The `ModulePath` component renders the lesson list as a visual path with zigzag layout. Each lesson node shows its completion status via colored ribbons:

- **Locked** — grayed out with lock icon
- **Default** — available to start
- **Complete** — green check
- **Mastered** — gold ribbon

The `FloatingTriggerButton` provides quick access to the module navigator on mobile via a slide-over panel.
