## Overview

The **admin app** (`apps/admin/`) is a separate application for managing Ludocode's educational content. It provides CRUD interfaces for courses, modules, lessons, exercises, languages, and subjects — all validated by Zod schemas with real-time visual error feedback.

---

## Admin Features

| Feature              | Route                                         | Description                               |
| -------------------- | --------------------------------------------- | ----------------------------------------- |
| **CoursesHub**       | `/hub/courses`                                | Lists all courses with create dialog      |
| **LanguagesHub**     | `/hub/languages`                              | Grid of language cards with create button |
| **Curriculum**       | `/curriculum/:courseId`                       | Module & lesson editor for a course       |
| **Lesson Builder**   | `/curriculum/:courseId/lesson/:lessonId`      | Exercise editor for a lesson              |
| **Language Manager** | `/language/:languageId` or `/language/create` | Create/modify programming languages       |
| **Subjects**         | `/subjects`                                   | Master-detail subject management          |

## Curriculum Editor

The curriculum editor is the heart of the admin app. It manages the module and lesson structure of a course using **curriculum draft snapshots**.

### Curriculum Draft Schema

```ts
export type CurriculumDraft = {
  modules: {
    id: string;
    title: string;
    lessons: {
      id: string;
      title: string;
    }[];
  }[];
};
```

The editor provides two views:

- **Preview mode** — read-only tree showing modules and their lessons
- **Edit mode** — full editor with drag-and-drop reordering, add/remove, and inline title editing

Modules can be reordered via `ModuleOrderActions`, and lessons within a module use `@dnd-kit/sortable` for drag-and-drop via `SortableLessonContainer`.

### Sidebar Detail Preview

Clicking a lesson in the tree opens a `LessonDetailPreview` in the right panel, showing exercise count and content preview.

## Lesson Exercise Builder

The lesson builder at `/curriculum/:courseId/lesson/:lessonId` manages the exercises within a single lesson.

### Exercise Draft Schema

Each exercise consists of **blocks** (content) and an optional **interaction** (answerable element):

```ts
export type CurriculumDraftLessonExercise = {
  exerciseId: string;
  exerciseVersion: number;
  blocks: CurriculumDraftBlock[];
  interaction: CurriculumDraftInteraction | null;
};
```

### Block Types

| Type        | Fields                | Purpose                        |
| ----------- | --------------------- | ------------------------------ |
| `header`    | `content`             | Section heading                |
| `paragraph` | `content`             | Explanatory text               |
| `code`      | `language`, `content` | Syntax-highlighted code sample |
| `media`     | `src`, `alt`          | Image/media embed              |

### Interaction Types

| Type     | Fields                          | Description                   |
| -------- | ------------------------------- | ----------------------------- |
| `SELECT` | `items[]`, `correctValue`       | Multiple choice               |
| `CLOZE`  | `file`, `blanks[]`, `options[]` | Fill-in-the-gap code exercise |

The builder provides separate editors for blocks (`BlocksEditor`) and interactions (`InteractionEditor`), both with preview and edit modes. Exercises can be reordered using drag-and-drop.

## Validation & Error Bubbling

All drafts are validated against Zod schemas. The admin uses a hierarchical validation approach:

1. **Exercise level** — validates blocks, interaction completeness, option counts
2. **Lesson level** — validates all exercises, checks for duplicates
3. **Module level** — validates all lessons, title requirements
4. **Course level** — validates all modules, minimum one module required

<Tip variant="tip" title="Error Propagation">
  An error in an exercise propagates upward — the exercise, its parent lesson,
  and the parent module all show red error indicators. This makes it easy to
  locate invalid content without manually expanding every node.
</Tip>

The admin uses a `StatusDot` component (green/red circle) as a visual indicator, and a validation error dialog with a Bell icon to show all current errors.

## Language Management

The language manager allows creating and modifying programming languages used across courses and projects.

```ts
export type LanguageMetadata = {
  languageId: number;
  name: string; // Display name (e.g., "Python")
  slug: string; // URL-safe identifier
  editorId: string; // Monaco Editor language ID
  pistonId: string; // Piston runtime ID
  extension: string; // File extension (e.g., ".py")
  base: string; // Base image
  iconName: string; // Icon registry key
  initialScript: string; // Default file content
};
```

The language form (`useLanguageForm`) provides field-level validation, slug uniqueness checking, and Monaco language autocompletion. The `useLanguageDiffs` hook tracks per-field changes for a visual diff display before saving.

---

## Subject Management

Subjects use a master-detail layout with the subject list on the left and a detail/edit form on the right. Subject slugs must be unique (enforced by Zod `superRefine`). The form uses `useSubjectForm` with `useSubjectDiffs` for change tracking.

---

## Forms & State

The admin app uses **TanStack Form** for the curriculum and exercise editors, providing:

- Type-safe form state derived from Zod schemas
- Field-level validation with `onSubmit` validators
- Custom field components (`FormTitleField`, `StatusDot`, `AddExerciseFieldButton`)

Language and subject forms use simpler `useState`-based forms with Zod validation on submit.
