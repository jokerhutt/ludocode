## Overview

Both the web and admin apps use **TanStack Query** for server-side data management. All server data â€” users, courses, lessons, streaks, projects, subscriptions â€” is stored under **query keys** and fetched through **query options**. Components read cached data reactively; they never hold server state themselves.

The data layer is structured into four files:

| File           | Purpose                                                                     |
| -------------- | --------------------------------------------------------------------------- |
| `qk.ts`        | Query key definitions â€” unique identifiers for each piece of data           |
| `queries.ts`   | Query options (`qo`) â€” binds keys to fetch functions with stale-time config |
| `mutations.ts` | Mutation options â€” POST/PATCH/DELETE operations with cache updates          |
| `batchers.ts`  | Batcher instances â€” groups individual ID lookups into batched requests      |

## Query Keys & Options

`qk` defines the cache keys. `qo` binds each key to a `queryOptions` object that describes how to fetch the data.

```ts
const qk = {
  streak: (userId: string) => ["streak", userId] as const,
};

const qo = {
  streak: (userId: string) =>
    queryOptions<UserStreak>({
      queryKey: qk.streak(userId),
      queryFn: () => ludoGet<UserStreak>(api.progress.streak.base, true),
      staleTime: 60_000,
    }),
};
```

To consume in a component:

```ts
const { data: streak } = useSuspenseQuery(qo.streak(userId));
```

`useSuspenseQuery` blocks rendering until data is available, eliminating loading/undefined checks in the component body.

## Batching

When a module contains many lessons, fetching each one individually would create excessive network requests. **Batchers** solve this by collecting individual ID lookups within a time window and sending them as a single request.

<Tip variant="info" title="How batching works">
  If the cache already has lessons `[1, 2, 5]` and the module needs `[1, 2, 3,
  4, 5]`, only `3` and `4` are sent to the backend. Cached items resolve
  immediately.
</Tip>

```ts
export const lessonBatcher = makeIdBatcher<LudoLesson>({
  name: "lesson",
  getUrlFn: api.lessons.lessons, // URL builder accepting comma-separated IDs
  idsKey: "lessonIds",
  scheduler: windowScheduler(10), // 10ms collection window
});
```

The query option uses the batcher as its fetch function:

```ts
lesson: (lessonId: string) =>
  queryOptions<LudoLesson>({
    queryKey: qk.lesson(lessonId),
    queryFn: () => lessonBatcher.fetch(lessonId),
  }),
```

The `makeIdBatcher` factory (from `@ludocode/api`) handles URL construction, deduplication, and result mapping automatically.

## Mutations

Mutations handle write operations. Each mutation hook wraps `useMutation` with appropriate cache updates in `onSuccess`.

### Cache Update Strategies

**Direct replacement** â€” when the server returns the updated object:

```ts
const qc = useQueryClient();
qc.setQueryData(qk.lesson(lesson.id), updatedLesson);
```

**Invalidation** â€” when the server doesn't return updated data:

```ts
qc.invalidateQueries({ queryKey: qk.weeklyStreak() });
```

## API Layer

All fetch calls go through typed wrappers in `@ludocode/api/fetcher.ts`:

| Function                 | HTTP Method | Notes                       |
| ------------------------ | ----------- | --------------------------- |
| `ludoGet<T>`             | GET         | Optional credential include |
| `ludoPost<TRes, TBody>`  | POST        | Handles 204 No Content      |
| `ludoPatch<TRes, TBody>` | PATCH       | Handles 204 No Content      |
| `ludoPut<TRes, TBody>`   | PUT         | Handles 204 No Content      |
| `ludoDelete<TRes>`       | DELETE      | Handles 204 No Content      |

All throw descriptive `Error` on non-OK responses. API URLs are built by `createApiPaths()`, which generates a structured object covering 15+ domains (auth, catalog, lessons, progress, runner, subscriptions, etc.).
